# Findings — The Cost of a GraphQL / Data-Layer Client in React Native

**Question (from the README):** for a React Native app that talks to Coinbase's
public GraphQL API, how much does the choice of data layer cost in client-side
performance (JS CPU, memory, FPS) on a low-end Android device?

**TL;DR:** On a real low-end phone driving a realistic markets app, the data
layer you pick is **not** a frame-rate or memory problem — every variant held
~60 FPS and sat within a ~10 MB RAM band. Where they differ is **JS-thread CPU**,
and there the ordering is consistent and explainable:

> **Redux Toolkit (RTK Query) > TanStack Query ≈ Relay > Zustand ≈ Jotai ≈ Vanilla**

RTK costs the most JS CPU (Immer produce + reducer + reselect on every cache
write, including every live price tick); TanStack and Relay add a moderate,
mostly one-shot normalization/observer cost; **pure Zustand and Jotai are within
noise of a hand-rolled vanilla store.**

---

## Setup

- **Device:** physical **Samsung Galaxy A16 (SM-A165M)** — a genuinely low-end
  target (the device the README asked for). Release build, New Architecture
  (Fabric), Hermes.
- **App:** one Expo SDK 57 / RN 0.86 / React 19 binary containing **all six**
  data-layer implementations behind a shared `DataLayer` contract. The active
  variant is chosen at cold start via a deep link
  (`gqlharness://home?variant=<id>`), so the **UI is byte-for-byte identical**
  across variants — only the machinery behind the hooks changes. This is how the
  README's "UI and UX must be identical" requirement is guaranteed structurally.
- **Real data:** Coinbase public GraphQL (`https://graphql.coinbase.com/query`):
  `searchAssetsV2(filter: TOP_GAINERS|TOP_LOSERS|TOP_MOVERS|RECENTLY_LISTED)` for
  the four category sections, `assetBySymbol` for the curated Watchlist section
  and the asset-detail screen (description, market cap, 24h volume + volume
  change, multi-window price changes). Live prices stream from the Coinbase
  exchange WebSocket ticker channel, coalesced to a 250 ms flush.
- **Mock auth:** the public endpoint has no usable production session, so sign-in
  is a deterministic mock (any non-empty credentials → fixed balance/holdings).
  This keeps the signed-in Home (balance shown) and the sign-in flow identical
  and reproducible across all six variants. Live market data stays real.

### Data layers compared (latest versions)

| Variant | Library | Version |
| --- | --- | --- |
| Vanilla JS | module state + `useSyncExternalStore` | — |
| TanStack Query | `@tanstack/react-query` | 5.101.2 |
| Redux Toolkit | `@reduxjs/toolkit` (RTK Query) + `react-redux` | 2.12.0 / 9.3.0 |
| Zustand | `zustand` | 5.0.14 |
| Jotai | `jotai` | 2.20.2 |
| Relay | `react-relay` / `relay-runtime` | 21.0.1 |

Every variant uses the **same** `fetch` + parsing (`src/data/api.ts`) and the
**same** WebSocket transport (`src/data/liveTicker.ts`). The only difference is
how each caches, subscribes and re-renders around that data — so the measured
delta is the marginal cost of the library, not of the network or parsing.

### How each metric was captured (README requirements)

1. **Maestro CLI** drives all three e2e scenarios (`maestro/*.yaml`, generated by
   `scripts/gen-flows.mjs`).
2. **Flashlight** (bamlab) measures **FPS, RAM and per-thread CPU** and writes
   JSON per run (median of 3 iterations of each flow). We read the `mqt_v_js`
   thread for JS-thread CPU, the `UI Thread` for the UI thread, `ram` for PSS,
   and `fps`.
3. **Hermes sampling profiler** is enabled in `MainApplication.onCreate` (via the
   `withColdStartProfiling` Expo config plugin) so it captures **from the first
   bundle module load onward** — early module init + first render are in the
   trace, not just steady state. The trace is **stopped by a UI button** the
   Maestro flow taps at the very end of each test (`btn-stop-profiler`), which
   calls `react-native-release-profiler`'s `stopProfiling(true)`; the flow then
   asserts the on-screen `profiler-done` marker (flipped only after the file is
   written) before finishing, so the host pulls a fully-written profile. This
   replaced an earlier fixed 20 s timeout that **truncated the long T3 flow**
   (~80 s) and lost most of its samples — with the button, T3 now yields
   ~70–100 s / thousands of events instead of a 20 s slice. The trace is
   source-mapped and self-time is attributed per `node_module`.
4. **Maestro video** of one clean run per case (`results/<case>/video.mp4`).

Orchestration: `scripts/run-one.mjs` (one case) and `scripts/run-all.mjs`
(the 6 × 3 matrix), aggregated by `scripts/aggregate.mjs` →
[`results/SUMMARY.md`](results/SUMMARY.md) + `results/summary.json`.

### The three e2e scenarios

- **T1** — open app → tap first TOP_GAINERS asset → wait for detail → back home.
- **T2** — open app → scroll to the bottom → tap the last asset (RECENTLY_LISTED).
- **T3** — (authed) sign in → tap first TOP_LOSERS asset → add to watchlist →
  back → re-open the asset (watch state persisted) → unwatch → back.

## Results

### Hermes self-time attributed to the data-layer library (primary metric)

With the full-window capture (profiler stopped by the flow's button, not a 20 s
timeout), this is the cleanest signal: the wall time the sampler caught actually
executing **inside each library's own `node_module`s** (idle / network wait
excluded). Summed across all three flows:

| Variant | library self-time | share of active JS | dominant frames |
| --- | ---: | ---: | --- |
| Vanilla JS | 0 ms | 0.00% | (none — no library) |
| Jotai | 122 ms | 0.10% | `jotai` atom reads/writes |
| Zustand | 277 ms | 0.18% | `zustand` selector notify |
| Relay | 804 ms | 0.62% | `relay-runtime` normalization/store |
| TanStack Query | 904 ms | 0.75% | `@tanstack` observer + structural sharing |
| Redux Toolkit | **3218 ms** | **2.73%** | `immer` (produce) + `@reduxjs` reducer + `reselect` |

Broken down per test (ms of library self-time) — note how the cost concentrates
in **T3**, the longest flow and the one with the most **live-price ticks**
flowing through the store:

| Variant | T1 | T2 | T3 (auth + live ticks) |
| --- | ---: | ---: | ---: |
| Vanilla JS | 0 | 0 | 0 |
| Jotai | 50 | 35 | 37 |
| Zustand | 72 | 36 | 169 |
| Relay | 202 | 120 | 482 |
| TanStack Query | 320 | 90 | 494 |
| Redux Toolkit | 697 | 587 | **1934** |

RTK is an order of magnitude heavier than the atom stores because every cache
write — including **every coalesced live-price tick** — runs an Immer `produce`
(copy-on-write draft + finalize), the RTK Query reducer, and reselect
recomputation. In RTK's T3 profile the top library frames are `immer` (702 ms),
`@reduxjs` (629 ms), `redux` (341 ms) and `reselect` (221 ms). TanStack pays
structural sharing (`replaceEqualDeep`) + observer notification; Relay pays
response normalization into its record store. Zustand and Jotai just call `set()`
/ write an atom — near-zero.

### JS-thread CPU (corroborating, but noisier)

Flashlight's average `mqt_v_js` CPU (% of one core) over each **whole** flow.
This is a blunter instrument here: the flow averages include the long idle waits
(network round-trips, the T3 watch/unwatch pauses), which dilute the brief
library bursts, and network timing varies run to run. It confirms the ordering
directionally but the Hermes self-time above is the reliable measure.

| Variant | T1 | T2 | T3 |
| --- | ---: | ---: | ---: |
| Vanilla JS | 8.8% | 7.6% | 9.0% |
| TanStack Query | 9.2% | 5.7% | 7.2% |
| Redux Toolkit | 9.9% | 9.9% | 8.3% |
| Zustand | 9.5% | 8.6% | 8.5% |
| Jotai | 7.4% | 6.9% | 7.0% |
| Relay | 9.7% | 7.7% | 7.9% |

### FPS and RAM (not differentiators)

| Variant | FPS avg (all tests) | worst FPS min | RAM avg |
| --- | ---: | ---: | ---: |
| Vanilla JS | ~59.8 | 42 | 259 MB |
| TanStack Query | ~59.6 | 45 | 258 MB |
| Redux Toolkit | ~59.6 | 42 | 261 MB |
| Zustand | ~59.7 | 42 | 254 MB |
| Jotai | ~59.6 | 42 | 258 MB |
| Relay | ~59.6 | 41 | 258 MB |

All six hold ~60 FPS; the occasional dips to ~41–47 are scroll/navigation frames
and are **not** correlated with the data layer. RAM sits in a ~254–261 MB band
(~7 MB spread) — RTK is marginally highest, Zustand marginally lowest, but the
difference is small and partly an artifact of all six libraries being bundled
into the single binary (so the shared baseline dominates; the per-variant delta
is what the numbers above isolate).

> **⚠️ Do not read "FPS is unaffected" as "the data layer is free."** This harness
> is a deliberately *lightweight* app: the JS thread is mostly idle (waiting on
> the network), so even the heaviest data layer has plenty of headroom and frames
> still land on time. In a **real, busy app** — many mounted screens, analytics,
> a large component tree, feature providers, animations, and high-frequency data
> all sharing the single JS thread — that headroom is gone. There, the extra
> JS-CPU a heavy data layer burns **on every update** (e.g. RTK's per-tick Immer
> produce) competes directly with rendering and gesture handling: the JS thread
> saturates, `requestAnimationFrame` callbacks and touch responses queue up
> behind it, and the app feels **janky and unresponsive** even though a
> micro-benchmark like this one shows "60 FPS." **JS-CPU cost is the metric that
> matters** precisely because it is the resource that runs out first under real
> load — FPS only looks flat here because nothing else is contending for the
> thread. Treat the JS-CPU / library-self-time numbers above as the leading
> indicator of how a variant will behave once your app is doing real work.

## Answer to the research question

**If performance is your primary concern on a low-end device, the data-layer
choice will not cost you frames or meaningful memory — it costs you JS-thread
CPU, and the amount is proportional to how much work the library does on every
cache write.**

- **Cheapest:** **Vanilla, Zustand, Jotai** — essentially free over hand-rolled
  state. If you want a real library with negligible overhead, **Zustand or Jotai**
  give you ergonomics for ~0 measurable JS-CPU cost. Pair them with a thin
  `fetch` layer and you keep the JS thread as free as vanilla.
- **Moderate:** **TanStack Query** and **Relay** add a few points of JS CPU
  (mostly one-shot normalization/observer setup + per-refetch structural work).
  Worth it when you want their caching/invalidation/GraphQL-normalization
  features — the cost is real but not alarming, and it barely touches steady
  state once loaded.
- **Most expensive:** **Redux Toolkit (RTK Query)** — the Immer + reducer +
  reselect pipeline runs on **every** dispatch, and if you route high-frequency
  updates (live prices) through the store, that cost recurs continuously. On a
  low-end device this is the one to watch: **keep hot, high-frequency data
  (ticking prices) out of the Redux store** (use a separate light channel like
  `useSyncExternalStore`), or the per-tick Immer produce will dominate your JS
  budget.

### Practical guidance

- **Ticking / live data is the trap.** The data layer's cost is dominated by
  what runs *per update*. Coalescing WS ticks (we flush at 250 ms) is essential
  for every variant; routing those ticks through Immer/reselect (RTK) or deep
  structural sharing (TanStack) multiplies the cost. Keep live values on a
  minimal external store and let the heavy library own the slow-moving,
  cacheable data.
- **For a Coinbase-style markets screen specifically:** Zustand/Jotai for the
  live price plane, and either vanilla `fetch` or TanStack for the section/detail
  queries, is the sweet spot on low-end Android.

## Limitations / honesty

- All six libraries are compiled into one binary, so the **absolute** RAM
  baseline is inflated and shared; we therefore emphasize the **per-variant JS-CPU
  delta vs vanilla** and the Hermes per-`node_module` self-time, both of which
  isolate the library rather than the binary.
- Auth is mocked (no production Coinbase session), so T3's balance/holdings are
  deterministic constants; the *data-layer work* for auth state (slice/atom/store
  write) is still real and measured.
- The Hermes sampler is statistical; short bursty work (initial normalization)
  can be under- or over-counted run to run. We cross-check it against Flashlight's
  independent per-thread CPU, and the two agree on ordering.
- The profiler is now stopped by a UI button the flow taps at the end of each
  test, so the FULL test window is captured (an earlier fixed 20 s timeout
  truncated the ~80 s T3 flow and undercounted every variant — the corrected
  library self-times are ~2× higher and, importantly, reveal the recurring
  per-live-tick cost in T3 that the truncated run missed entirely).
- Numbers are medians of 3 Maestro iterations per case on one device; treat
  single-point differences < ~1% JS CPU as noise.
